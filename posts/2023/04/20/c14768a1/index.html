概述

自己对与评论系统的理解，也多亏在折腾博客的过程中逐渐有清晰的认识

以前以为文章、论坛的帖子下面的评论模块都是开发者写的，那不折腾死？后来发现是有专门的评论系统，只需要嵌入到自己的文章内就好了

对于有后台的博客系统来说，有自带的评论系统，如wordpress；而自己的是静态的，没有后台管理的功能，文章的管理都需要从源代码层面进行修改，然后重新生成进行部署，评论系统也是借用第三方的

期望的

支持账号登录之余，能匿名评论，文章的评论能进行管理审核

折腾过

gittalk

https://gitalk.github.io/

基于github issue的功能，实现对文章的评论讨论，毕竟issue在github中也是用于反馈代码问题，所以这么看，用作评论也是可以的

个人觉得不太友好的点

  • 需要github账号，并需要登录才能评论
    可能这个评论系统的初衷是面向技术人员开发的，基于github的功能，所以没办法

  • 文章的评论是分散的
    每一个评论都是独立的issue，不能以文章为基准把所有相关的评论都集中到同一个issue中；
    没办法管理评论，无法删除，只能close

livere

https://www.livere.com

支持多个社交系统的登录

也有评论审核机制

使用很简单，在代码管理找到uid即可

总体来说livere已经是不错的选择了，但是我一直比较在意强制登陆才能评论的点，用了一段时间也放弃了

waline

https://waline.js.org/
https://waline.js.org/guide/get-started.html

更换Keep主题后，目前在使用的，完全符合自己的期望；唯一的就是搭建比较麻烦，需要折腾一下

目前对这个系统已经很满意了，暂时不想进行替换；唯一担心的点是以后对于评论数据的迁移

折腾：主要通过 leancloud+github+vercel+waline

  • leancloud（国际版），用它的数据库进行存储评论数据
  • github repo 存放部署的代码
  • vercel 部署工具
  • waline 评论系统

waline部署成功后，登陆的后台管理

嵌入效果，与Keep的简约主题进行了些许的调整

总结

评论系统个人认为是静态博客的两大“烦”点之一，另一个是图床；

因为这两个数据都放在第三方的服务上，以后是否会出幺蛾子，对于个人数据保护和迁移难度；都是需要考虑的

就想折腾到这了 👀
